# DESIGN.md — 결정론적 웹 멀티 아케이드 프레임워크

> 이 문서는 프로젝트의 아키텍처, 게임 플로우, 그리고 반드시 지켜야 할 설계 불변식을 정의한다.
> 코드를 작성하기 전에 이 문서를 읽고, 여기 적힌 원칙과 충돌하는 구현은 하지 않는다.

---

## 0. 한 문장 요약

같은 시드로 모두가 같은 세계(화살 패턴 등)를 각자 화면에서 겪고, 게임 판정은 결과(생존/사망)만 동기화하는
**결정론적 동기화 기반 웹 멀티 아케이드 프레임워크**. 관전 품질을 위해 플레이어 위치는 10Hz 스냅샷으로 별도 중계한다.
그 위에 게임을 여러 개 얹는다. 첫 게임은 "죽림고수"(화살 피하기).

핵심 설계 철학: **설계 판단은 사람이, 반복 검증은 AI가.**

---

## 1. 기술 스택

- **언어**: TypeScript (클라이언트·서버·공유 계약 전부)
- **렌더링**: Canvas 2D로 시작. `IRenderer` 인터페이스로 추상화하여, 필요 시 PixiJS(WebGL)로 교체 가능하게 둔다.
- **클라이언트 게임 루프**: 고정 타임스텝 + **rAF + `setInterval` 폴백** (아래 참조)
- **네트워크**: WebSocket (`ws` 라이브러리). Photon 아님. WebGL/브라우저 표준.
- **서버**: Node.js + `ws`. 작은 authoritative 서버.
- **빌드 타깃**: 웹 (링크 클릭 → 바로 플레이). 정적 클라 + WebSocket 서버 분리 배포.
- **구조**: 모노레포 (shared / core / games / server / app)
- **테스트**: Vitest (`npm test`) — 결정론·클럭·FSM·방 흐름·서버 통합 회귀

### 왜 rAF만으로는 안 되는가 (GameLoop)

`requestAnimationFrame`은 **탭이 백그라운드로 가면 브라우저가 멈춘다.** 싱글이면 상관없지만
멀티에서는 치명적이다 — 다른 탭을 보다 돌아오면 내 월드만 몇 초 뒤처져 있고, 남들은 이미
저만치 진행해 있다. 결정론이 깨진 게 아니라 **내 tick이 안 흐른 것**이지만 결과는 똑같이 망가진다.

그래서 `GameLoop`는 두 축으로 돈다:
- **rAF**: 화면이 보일 때의 부드러운 렌더 (자유 프레임레이트, 보간 alpha 사용)
- **`setInterval` 폴백**: 백그라운드에서도 tick을 계속 전진시킨다

둘 다 "예약된 시작 시각(epoch)부터 흐른 실시간"에서 tick을 계산하므로 **중복 전진하지 않는다.**
tick의 유일한 근거는 벽시계 경과 시간이지, 콜백이 몇 번 불렸는지가 아니다.

> 왜 Unity가 아닌가: 원래 만들고 싶었던 것이 TETR.IO 같은 "웹이 곧 게임"인 웹 네이티브 경험이기 때문.
> C#/Unity 아키텍처 감각(IRepository, FSM, 오브젝트 풀링)은 TypeScript로 1:1 이식된다.

## 2. 결정론 불변식 (THE INVARIANT — 절대 규칙)

이 프로젝트의 심장. 이 규칙이 깨지면 클라이언트마다 화살이 달라져 게임이 무너진다.

> **게임 결과에 영향을 주는 모든 것은 시드와 tick에서만 파생된다.**
> **시스템 시계·전역 상태·실측 delta는 결과 계산에 절대 끼어들지 않는다.**

구체 규칙:

1. **전역 난수 금지.** `Math.random()`을 게임 로직에 쓰지 않는다. 시드로 초기화한 `SeededRNG`(mulberry32)
   인스턴스를 스포너 등이 소유한다. 시각 효과(파티클·화면 흔들림 등 게임 결과에 무관한 것)에만 `Math.random()` 허용.
2. **고정 타임스텝.** 게임 로직 업데이트는 항상 고정 간격(1/60초)으로만 전진한다. `update(tick)`은 실측
   `deltaTime`을 받지 않는다. 렌더는 자유 프레임레이트로 하되 보간(alpha)만 사용.
3. **난이도·스폰 등 모든 진행은 tick의 함수.** `Date.now()`로 난이도를 올리지 않는다. 예: `difficulty = floor(tick / 300)`.
4. **판정은 로컬, 동기화는 결과만.** 피격 판정은 각 클라이언트가 자기 화면에서 내린다. 부동소수점 완전 결정론까지
   갈 필요 없음 — 화살 패턴만 시각적으로 동일하면 충분. 네트워크로는 "죽음/생존시간"만 전송.
5. **실시간 epoch 기반 따라잡기.** tick은 예약 시작 시각 이후 흐른 실시간에서 계산한다. 백그라운드에서
   놓친 tick은 한 번에 최대 300개씩 여러 호출에 나눠 따라잡아 AFK로 월드가 뒤처지거나 UI가 오래 멈추지 않게 한다.

> "어디까지 결정론으로 맞추고 어디부터 로컬에 맡길지 선을 긋는" 것이 핵심 설계 판단 지점이다.

## 3. 아키텍처 — 계층과 의존성 방향

### 의존성 규칙 (OPEN-CLOSED)

> **모든 의존성 화살표는 위로만 향한다. 코어는 게임을 모른다.**

- `shared` (계약: IGame, 메시지 타입) ← 모두가 의존
- `core` (엔진: 루프·RNG·네트워크·FSM) ← games, server가 의존. **games를 import하지 않는다.**
- `games/*` (게임별 규칙) → core, shared에 의존
- `server` → shared에 의존 (게임 내용을 모름. gameId 문자열만 중계)
- `app` (진입점) → 게임들을 GameRegistry에 등록

새 게임 추가 시 core는 한 줄도 바뀌지 않아야 한다. 이 불변이 깨지면 추상화가 새는 것이다.

### 폴더 구조

```
WebArcade/
├─ packages/

│  │  ├─ IRenderer.ts      # 렌더 계약 (⚠️ core가 아니라 여기. 아래 근거 참조)
```

**왜 `IRenderer`가 `core`가 아니라 `shared`에 있는가?**
`games/*`가 렌더를 하려면 렌더 계약을 알아야 한다. 그 계약이 `core`에 있으면 게임이 엔진 구현체에
묶인다. 계약(`IRenderer`)은 `shared`에, 구현(`Canvas2DRenderer`)은 `core`에 두면 게임은 **"무엇을
그릴지"만 알고 "어떻게 그리는지"는 모른다.** PixiJS로 갈아끼워도 게임 코드는 한 줄도 안 바뀐다.
`IGame`이 게임 → 엔진 방향의 계약이라면, `IRenderer`는 엔진 → 게임 방향의 계약이다. 둘 다 `shared`.

### IGame 계약 (잠정 — 두 번째 게임이 검증할 때까지 확정하지 않음)

```typescript
export interface IGame {
  init(seed: number): void;                       // 시드로 결정론 초기화
  update(tick: number, input: InputState): void;  // 고정 스텝 전진
  render(r: IRenderer, alpha: number): void;       // 자기 렌더
  renderSpectator(r: IRenderer, target: SpectateTarget): void;
  isPlayerDead(): boolean;                         // 로컬 사망 판정
  getPosition(): { x: number; y: number };         // 관전 위치 전송
  syncPeers(peers: readonly PeerState[]): void;    // 관전 위치 반영
  getScore(): number;                              // 순위 기준
}
```

재시작은 `init(newSeed)` 재호출로 통일한다. 관전 위치와 개인 조준 화살은 시각적 근사이며 판정에는 사용하지 않는다.

**잠정적으로 열어둘 지점** (두 번째 게임에서 실제로 부딪히면 넓힌다. 상상으로 미리 넓히지 않는다):
- ~~`getScore()`의 우열 방향~~ → **해소됨.** `ScoreDirection`('higher'|'lower') 타입으로 분리하여
  게임 레지스트리 항목(`scoreDirection`)에 담는다. 죽림고수='higher'. (친구 gahee의 데이터 분리 패턴 차용)
- 입력이 4방향으로 충분한지 (회전·하강 등 필요한 게임 대비) — `InputState`는 아직 4방향 고정
- 종료 조건이 "죽음" 하나인지 (목표 도달로 끝나는 게임 대비)

### GameRunner — 코어가 게임을 모른 채 구동하는 방식

`GameRunner`는 `JungnimGame`이라는 단어를 절대 쓰지 않는다. 오직 `IGame` 메서드만 호출한다.
이것이 IRepository로 데이터 계층을 추상화한 것과 동일한 의존성 역전이다.

## 4. 게임 전체 플로우

접속 → 순위표 → 재시작까지의 라이프사이클. (사용자가 직접 설계한 플로우)

1. **접속 · 닉네임**: 사이트 진입 시 닉네임을 한 번 정한다. 안 정하면 랜덤 배정. 중복 허용(순위표 표시용).
   URL에 방 코드(`#WXYZ`)가 있으면 자동 입장 시도.
2. **메인 · 게임 목록**: 게임 시작 / 옵션 / 크레딧. "게임 시작" → 게임 목록(GameRegistry에서 자동 생성).
3. **로비**: 게임 선택 후 진입. 실시간 대전 찾기 / 방 만들기 / 방 코드 입력.
   - **실시간 대전 찾기는 나중에 구현** (매치메이킹은 별도 서브시스템. 코드 기반 방을 먼저 완성).
4. **방 대기실 (Ready)**: 방 만들기 → 코드 발급(I·O 제외 4자리). 코드로 참가. v1 최대 32명.
   - 준비 버튼 없음. **호스트가 시작 버튼을 누르면 즉시 시작** (TETR.IO 방식).
   - 호스트가 나가면 **남은 사람 중 가장 먼저 들어온 사람**에게 이양(서버가 입장 순서 리스트로 관리).
5. **시드 배포 · 카운트다운**: 연결 시 5회 ping/pong 중 RTT가 가장 낮은 표본으로 서버 시각을 보정한다.
   호스트 시작 → 서버가 `game_start(seed, startTime)` 브로드캐스트. 각 클라는 보정된 시각과 예약 epoch로
   카운트다운하고, 늦은 콜백도 예약 시각부터 경과한 tick을 따라잡는다.
6. **플레이**: 본인 화면 크게 + 우측에 무작위 생존자 최대 3명.
   - 화살은 각 클라가 시드로 로컬 생성. 피격 판정도 로컬.
   - 관전 대상이 죽으면 다른 무작위 생존자로 교체.
   - **레이아웃 축소 규칙(공통)**: 항상 "생존자 중 최대 3명"을 보여주되 부족하면 있는 만큼만. 1+3 → 1+2 → 1+1.
     이 규칙은 플레이 중이든 관전 중이든 동일하게 적용된다.
7. **본인 사망 시 선택지**:
   - (a) 결과창 유지 — 메인 자리에 내 결과, 우측에 관전 대상.
   - (b) 관전 — 메인 자리도 남의 화면. 관전 1명 + 관전 최대 3명(1+3→1+1 축소 규칙 적용).
8. **게임 종료**: 전원 사망 시. **상태와 무관하게 그 자리에서 순위표 표시**
   (관전 중이면 관전 화면 위에, 생존자면 플레이 화면 위에. "본인 화면 복귀" 없음).
   - 순위표 = **등수 + 닉네임 + 생존시간** (생존시간 긴 순).
9. **순위표 다음**: 같은 방 다시 하기(새 시드로 재시작 → 대기실) / 로비로 가기(방 나가기).

> **관전 대상 선택과 레이아웃은 로컬 판단이다.** 서버는 게임 화면을 시뮬레이션하지 않고, 각 클라이언트가
> 보낸 최신 위치를 방 단위 10Hz `peer_snapshot`으로 묶어 중계한다. 공통 화살은 seed/tick으로 정확히
> 재현하고, 원격 아바타와 개인 조준 화살은 위치 스냅샷을 보간한 시각적 근사다.

## 5. 네트워크 프로토콜 (shared/protocol.ts)

```
클라 → 서버:  join_room(code, nickname) · create_room(gameId, nickname) · start_game
             time_sync_request · player_state(px, py) · player_died(survivalTicks)
             return_to_ready · leave_room
서버 → 클라:  welcome(id) · time_sync_response · room_state(state, players, hostId)
             game_start(seed, startTime, gameId) · peer_snapshot(peers) · peer_died(id)
             ranking_update(alive, ranks) · game_over(finalRanks) · host_changed(newHostId)
```

- 서버는 게임 내용을 모른다. `gameId`는 문자열로 중계할 뿐.
- 화살은 네트워크를 한 번도 타지 않는다. 관전 위치만 플레이어당 10Hz로 보내며 서버가 방별 스냅샷으로 묶는다.
- v1은 최대 32명으로 검증한다. 그 이상 확장은 측정 후 전송 빈도·압축·관전 정책을 다시 결정한다.

## 6. 서버 — 방 관리 (결정론 덕에 단순함)

서버가 들고 있는 것: 방 코드, 접속자(입장 순서 리스트), 상태, 시드, 시작 시각, gameId, 최신 관전 위치.

- `Room`: 접속자를 **순서 있는 리스트**로 관리(호스트 이양의 근거). 최대 32명, 진행 중이면 입장 거절.
- 진행 중 연결 종료는 서버 경과 tick의 사망으로 기록하고 해당 라운드 최종 순위에는 남긴다.
- `RoomManager`: 코드 생성 시 I·O 제외한 문자셋 사용. `start()`에서 시드 + `startTime = now + 3000` 배포.
- `RankingService`: 생존 신호 수집, 순위 계산, `ranking_update` 중계.

**나중에 정할 것 (지금 안 정해도 됨)**:
- 호스트가 나가서 방이 빌 때 처리 / 진행 중 입장을 관전으로 허용할지 / 인원 많을 때 UI 표시 기준.

## 7. 개발 순서 (반드시 이 순서)

추상화는 상상으로 하지 않는다. 만들고 나서 두 번째 사례로 검증하며 추출한다.

1. ✅ **코어 + shared(IGame) + games/jungnim을 처음부터 분리된 구조로 짠다.**
   죽림고수도 처음부터 `implements IGame`. 단 인터페이스는 잠정.
2. ✅ **죽림고수 싱글 완성** — GameLoop·SeededRNG·FSM·Canvas 렌더 검증. 혼자 플레이되는 죽림고수.
3. ✅ **죽림고수에 멀티 부착, 전체 플로우를 끝까지 돌린다** — 방·시드 동기화·순위·관전.
   ⚠️ 싱글만 돌 때 두 번째 게임을 붙이지 않는다. 멀티까지 전체 플로우가 한 바퀴 돌아야 코어가 검증된다.
4. ✅ **멈추고 훑고 청소한다** — core에 죽림고수 냄새(예: GameRunner가 "화살"을 안다)가 새어들었는지 점검·제거.
   여기서 나온 것: 범용 `StateMachine` 추출, app을 AppFlow/AppView/GameSession으로 분해,
   서버를 `ArcadeServer` + `validation`으로 분리, Vitest 회귀 스위트.
5. ⬅️ **[현재] 두 번째 게임을 붙이며 IGame이 안 맞는 부분을 넓힌다** — 이 시점에 추상화가 진짜로 확정된다.
6. **세 번째부터**는 IGame 구현 + GameRegistry 등록만.

> 3~4단계 이후 배포(9절)까지 마쳐 링크 하나로 플레이 가능한 상태다. 다만 **결정론이 진짜
> 네트워크·기기 차이를 견디는지는 아직 다수 동시 플레이로 검증되지 않았다.** 5단계 전에 이걸
> 먼저 확인하는 편이 낫다 — 코어가 틀렸다면 두 번째 게임은 틀린 코어 위에 얹히기 때문이다.

## 8. 외부 참고 (gahee 프로젝트에서 채택한 것)

친구의 웹 프로젝트(`eddy0626/gahee`, React+TS+Vite 정적 마케팅 사이트)에서 실제로 확인하고 채택한 것들.
게임 로직·멀티는 참고할 게 없었지만, 웹 세팅과 데이터 확장 패턴에서 가져왔다.

- **tsconfig 옵션 세트**: `strict`, `noUnusedLocals`, `noUnusedParameters`, `noFallthroughCasesInSwitch`,
  `forceConsistentCasingInFileNames`. (실제로 이 규칙이 뼈대 단계에서 안 쓰인 필드를 잡아냄 → 검증됨)
- **빌드 게이트**: `"build": "tsc -b && vite build"` — 빌드 전 타입 검사를 강제.
- **문자열 유니온 타입으로 오타 차단**: 친구의 `Platform` 유니온 → 이 프로젝트의 `GameId`(레지스트리 키 유니온),
  `ScoreDirection`. 존재하지 않는 값 참조를 컴파일 타임에 막는다.
- **데이터 자동 확장**: 친구의 `content.ts` `games` 배열(추가 시 UI 자동 확장) → 이 프로젝트의 `GameRegistry`
  (등록 시 게임 목록 자동 생성). 차이: 친구 건 "보여줄 데이터", 이건 "실행할 게임"(factory 포함).
- **게임별 설정을 데이터로 분리**: 친구가 게임 속성을 `Game` 타입에 담듯, 죽림고수 튜닝값을 `config.ts`에 분리.
- **문서화 습관**: 친구의 README "게임 추가법" 섹션 → 이 프로젝트 README에도 "새 게임 추가법" 명시.

채택하지 **않은** 것: React(게임 화면은 Canvas 중심 — UI 필요 시 나중에 추가), 다국어·문의 폼·법률 페이지(마케팅 전용).

## 9. 배포 (완료 — 라이브)

"완결"에는 배포까지 포함한다. 링크 하나로 남이 플레이할 수 있어야 한다.

- **프론트(정적)**: Vercel — https://web-arcade-sigma.vercel.app
- **게임 서버(WebSocket)**: Fly.io(도쿄/nrt) — `wss://webarcade.fly.dev`

### 왜 둘을 따로 올리는가

성격이 정반대다. 프론트는 **한 번 넘겨주고 끝나는 정적 파일**(index.html + JS 한 덩어리)이고,
서버는 **연결을 계속 붙들고 있는 상시 프로세스**다. 정적 파일은 CDN이 공짜로 잘 뿌리고, 상시
프로세스는 CDN이 못 한다. 섞으면 양쪽 다 나빠진다.

게임은 **서버에서 돌지 않는다.** 각자 브라우저가 같은 시드로 같은 게임을 돌리고, 서버는 "시드는
X다", "이 사람 죽었다", "위치는 여기다" 같은 짧은 사실만 중계한다. 그래서 256MB 서버 1대로 충분하다.

### ⚠️ 서버 머신은 반드시 1대

방 상태가 `RoomManager`의 **인메모리 `Map`** 이다(DB 없음). 여기서 두 가지가 따라온다:

1. **머신이 2대면 Map도 2개다.** 각 프로세스가 자기 Map을 갖고 서로 대화하지 않으므로, 같은 방
   코드를 쳐도 다른 머신에 붙은 사람은 "그런 방 없음"을 받는다. 누가 어느 머신에 붙을지는 운이라
   "될 때도 있고 안 될 때도 있는" 최악의 버그가 된다.
2. **프로세스가 재시작하면 방이 사라진다.** `fly deploy`는 서버를 재시작한다 → **사람들이 플레이
   중일 때 배포하면 전원 튕긴다.**

`fly.toml`이 `auto_stop_machines = 'off'` / `min_machines_running = 1`로 고정해 뒀다.
스케일아웃이 필요해지는 날에는 **방 상태를 Redis 등 외부 저장소로 빼는 것이 선행 조건**이다.

### ⚠️ `VITE_WS_URL`은 빌드 타임에 박힌다

프론트가 붙을 서버 주소는 Vite가 **번들에 문자열로 굳혀 넣는다.** 런타임에 읽는 값이 아니다.
- 서버 주소가 바뀌면 환경변수만 고쳐선 소용없다 → **`vercel --prod`로 다시 빌드해야** 반영된다.
- 페이지가 https이므로 반드시 `wss://`. `ws://`는 브라우저가 mixed content로 차단한다.

### 배포 시 재빌드 범위

바꾼 패키지에 따라 어느 쪽을 다시 올릴지 갈린다. **`shared`(프로토콜)를 고치면 반드시 둘 다.**
한쪽만 올리면 서버와 클라가 서로 다른 메시지 형식을 쓰게 되어 조용히 깨진다.

| 고친 곳 | 재배포 |
|---|---|
| `app`, `core`, `games` | `vercel --prod` |
| `server` | `fly deploy` |
| `shared` | **둘 다** |

## 10. 하지 않기로 한 것 (그리고 왜)

코드에는 **없는 것**이라 주석으로 남길 자리가 없다. 하지만 "왜 이건 안 만들었지?"는
반드시 다시 떠오르므로 여기 적는다. 각 항목에 **재검토 조건**을 함께 둔다 —
"영원히 안 함"과 "아직 안 함"은 다르다.

### 서버 권위 판정 / 리플레이 검증 — 하지 않는다

점수는 클라이언트가 자기 `survivalTicks`를 **자기신고**한다. 서버는 상한만 검증한다
(`ArcadeServer.ts`의 `SURVIVAL_TOLERANCE_TICKS` — 방 경과 시간보다 오래 버텼다는 주장은 거부).
**일찍 죽고 늦게 신고하는 것은 막지 못한다.**

역설적이게도 이 프레임워크는 결정론이라 **원리적으로 완벽한 검증이 가능하다** — 시드와 입력
시퀀스를 받아 서버가 재생하면 진위를 가릴 수 있다. 그런데 그러려면 **서버가 게임 로직을 돌려야
하고**, 그 순간 "서버는 게임을 모른다"(6절)와 의존성 방향(3절)이 동시에 깨진다. 치트 방지 하나로
아키텍처의 중심 원칙 두 개를 파는 거래는 하지 않는다.

→ **재검토 조건**: 실제로 조작된 기록이 관측될 때. 그전까지는 전역 랭킹을 **멀티 기록으로 한정**해
(같은 방에 증인이 있고 상한 검증이 걸린다) 회피한다. 솔로 기록은 개인 최고 기록으로만 둔다.

### 디싱크 감지 (월드 해시 대조) — 보류

각 클라이언트가 월드 상태를 해시해 서버가 비교하면 결정론이 깨진 순간을 잡을 수 있다.
만들다가 중단했다 — **증상이 없는 상태에서 만드는 진단 도구**였기 때문이다. 결정론 자체는
헤드리스 테스트로 이미 검증된다(공통 화살 452발 0.000000px 오차).

→ **재검토 조건**: "화살이 남과 어긋난 것 같다"는 실제 제보. 그때 만들면 이미 재현 사례가
있으므로 도구가 훨씬 정확해진다.

### 관전 대상 고르기 — 만들지 않는다

죽은 뒤 누구를 볼지 목록에서 고르는 UI. 의도적으로 뺐다. 대신 자동 전환 + ← → 키로 넘기기만
둔다. 사망 직후는 감정적으로 격한 순간이라 **선택지를 들이미는 것이 연출을 끊는다.**

→ 재검토 조건 없음. 설계 의도다.

### 게임 중 자동 재접속 — 만들지 않는다

끊긴 사람이 판에 복귀하는 기능. 결정론이라 기술적으로는 가능하다(시드와 시작 시각만 받으면
따라잡을 수 있다). 하지만 **이탈이 이미 무해하게 처리된다** — 남은 사람은 계속 플레이하고,
기록은 보존되며, 방장은 자동 승계되고, 빈 방은 60초 유예 후 정리된다. 복귀해도 이미 죽어 있다.

→ 재검토 조건: 게임 길이가 크게 늘어 중도 이탈의 손실이 커질 때.

### 서버 다중화 — 지금 구조로는 불가능

방 상태가 `RoomManager`의 `Map`, 즉 **프로세스 메모리**에만 있다. 2대 이상이면 같은 방 코드를
쳐도 다른 머신에 붙은 사람은 방을 못 찾는다(9절 경고 참조). 이건 "안 하기로 한 것"이 아니라
**구조적 제약**이다.

→ 해소 경로: Cloudflare Durable Objects로 옮기면 **방 하나 = 오브젝트 하나**가 되어 제약이
사라진다. 무료 티어 조건은 확인해 뒀다(들어오는 WebSocket 메시지에 20:1 청구 비율이 적용되어
10명 기준 하루 5.5시간). 착수는 현재 호스팅 크레딧이 소진될 때.

### 조기 추상화 — 금지 (7절 재확인)

`IGame`은 죽림고수 하나만 보고 만든 **잠정 계약**이다. 두 번째 게임이 실제로 부딪히기 전에는
넓히지 않는다. 이미 굽어 있을 것으로 의심되는 지점은 셋이다: `getPosition()`이 플레이어를
"점 하나"로 가정하는 것, `InputState`가 4방향 고정인 것, `PeerState {x, y}`로는 격자·보드류의
관전을 그릴 수 없는 것. **의심은 기록하되 고치지 않는다** — 두 번째 게임이 어느 쪽을 실제로
요구하는지 보고 정한다.
