# AGENTS.md — AI 협업 규칙

> 이 프로젝트에서 AI(Claude Code, Codex, Gemini 등)가 따라야 할 컨벤션과 협업 방식.
> `DESIGN.md`를 먼저 읽고, 이 문서의 규칙을 지켜 작업한다.

---

## 0. 협업 철학 (가장 중요)

**설계 판단은 사람이, 반복 검증은 AI가.**

이 프로젝트의 아키텍처와 게임 플로우는 개발자가 직접 설계했다. AI의 역할은 그 판단을 대신하는 것이
아니라, **검증하고 확장하는 조력자**다. 구체적으로:

- 설계 결정(무엇을 코어로, 무엇을 게임별로 나눌지 등)은 개발자에게 확인받는다. 임의로 바꾸지 않는다.
- 개발자가 방향을 제시하면, AI는 그 방향의 함정·트레이드오프·놓친 케이스를 짚어준다.
- "이렇게 하는 게 맞나요?"라는 질문에는, 정답을 단정하기보다 선택지와 각각의 결과를 제시한다.
- 애매하면 진행하기 전에 되묻는다. 특히 `DESIGN.md`의 불변식에 영향을 주는 변경은 반드시 확인.

> 좋은 결과물은 AI가 설계를 던져주고 사람이 수락해서가 아니라, 사람이 판단하고 AI가 검증해서 나온다.

---

## 1. 절대 규칙 (NEVER VIOLATE)

`DESIGN.md`의 두 축을 코드 레벨에서 다시 못박는다. 이 둘을 어기는 코드는 작성하지 않는다.

### 1-1. 결정론 불변식
- 게임 로직에서 `Math.random()` **금지**. `SeededRNG` 인스턴스만 사용. (시각 효과 예외)
- `update()`에 실측 `deltaTime`을 넘기지 **않는다**. 고정 스텝(1/60초)만.
- 난이도·스폰 등 진행은 `tick`의 함수. `Date.now()` 기반 진행 **금지**.
- 새 값을 추가할 때 자문: "이 값이 게임 결과에 영향을 주나? 그렇다면 시드/tick에서 나왔나?"

### 1-2. 의존성 방향
- `core`는 `games`를 **import하지 않는다**. 코어에 특정 게임 이름·개념(예: "화살")이 등장하면 안 된다.
- 게임별 로직은 `games/*`에만. 공통 계약은 `shared`에만.
- 새 게임 추가로 `core`가 수정되어야 한다면, 그건 추상화가 새는 신호 → 개발자에게 알린다.

## 2. TypeScript 컨벤션

- `strict: true`. `any` 지양. 불가피하면 주석으로 이유 명시.
- 네이밍: 클래스·인터페이스 `PascalCase`, 함수·변수 `camelCase`, 상수 `UPPER_SNAKE`, 파일은 클래스명과 일치.
- 인터페이스는 접두사 `I`를 붙인다(`IGame`, `IRenderer`) — 이 프로젝트의 추상 계약임을 드러내기 위해.
  (C#에서 넘어온 컨벤션을 의도적으로 유지. IRepository → IGame의 일관성.)
- 한 파일 = 한 주요 책임. 파일이 커지면 분리.
- 부수효과 없는 순수 함수 선호. 특히 게임 로직은 "같은 입력 → 같은 출력"이 결정론과 직결.
- 주석은 "왜"를 적는다. "무엇"은 코드가 말하게 한다. 특히 결정론 관련 코드엔 불변식을 주석으로 남긴다.

---

## 3. 아키텍처 준수 사항

- 새 게임: `games/<name>/`에 `IGame` 구현 + `config.ts`. `GameRegistry`에 등록. 그 외 파일 건드리지 않기.
- 렌더러 교체 가능성 유지: 게임은 `IRenderer`에만 의존하고 Canvas 2D API를 직접 호출하지 않는다.
- 오브젝트 풀: 대량 생성/소멸 객체(화살 등)는 `new`를 매 프레임 호출하지 않고 풀에서 재사용.
  풀의 acquire/release 순서도 tick 기반이어야 결정론이 유지됨.
- 상태 관리: FSM(`Ready → Countdown → Playing → (Dead) → Result`)으로. 임시 boolean 플래그 남발 금지.

## 4. 3-tier AI 워크플로

개발자의 기존 워크플로를 이 프로젝트에 맞게 적용한다.

- **Codex (AGENTS.md 기반)**: 이 문서의 컨벤션에 맞는 보일러플레이트·템플릿 생성.
- **Claude Code + Serena MCP**: 심볼 단위 아키텍처 분석. 특히 개발 순서 4단계("훑고 청소")에서
  `core`가 `games`를 import하는 지점, 코어에 새어든 게임 의존을 심볼 단위로 탐색·제거하는 데 활용.
- **Gemini Code Assist (styleguide 기반)**: 이 문서를 styleguide 삼아 PR 리뷰. 결정론 불변식·의존성 방향
  위반을 우선 점검 항목으로.

## 5. 작업 시 체크리스트

코드를 제출하기 전에 스스로 확인:

- [ ] 게임 로직에 `Math.random()`이나 `Date.now()` 기반 진행이 없는가?
- [ ] `update()`가 고정 스텝만 쓰는가? (실측 delta 미사용)
- [ ] `core`가 특정 게임을 import하거나 게임 개념을 알고 있지 않은가?
- [ ] 새 게임이 `IGame`만으로 구현됐는가? core 수정이 필요했다면 개발자에게 보고했는가?
- [ ] `IRenderer` 추상을 우회해 Canvas API를 직접 부르지 않았는가?
- [ ] 이 변경이 `DESIGN.md`의 결정 사항과 충돌하지 않는가? 충돌하면 진행 전 확인했는가?

---

## 6. 개발 단계 게이트

`DESIGN.md` 7절의 순서를 강제한다. 특히:

- **싱글이 완성되기 전에 멀티 코드를 섞지 않는다.**
- **멀티까지 전체 플로우가 한 바퀴 돌기 전에 두 번째 게임을 추가하지 않는다.**
- 두 번째 게임 추가는 IGame 확장이 정당한지 개발자와 합의한 뒤 진행.

이 게이트를 건너뛰자는 제안이 나오면, AI는 진행하지 말고 순서의 이유(검증되지 않은 조기 추상화 위험)를
상기시킨다.
